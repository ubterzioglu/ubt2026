Requirements Interview

Tek cümlelik bir fikri, kod yazılmadan önce yazılı bir spesifikasyona dönüştürür — projenin büyüklüğüne göre ölçeklenen bir soru grubuyla seni interview eder, sonra cevapları SPEC.md dosyasında toplar.

Why this exists

A one-line pitch hides a hundred unstated decisions. Left unstated, the agent fills them in with defaults — and the defaults are usually wrong in some detail you cared about.

The value isn't the question count. It's writing decisions down before code exists, so building doesn't quietly re-decide them one file at a time.

Neden var

Tek cümlelik bir fikir, yüzlerce söylenmemiş kararı içinde gizler. Söylenmezse, ajan bunları varsayılan değerlerle doldurur — ve bu varsayılanlar genellikle senin önemsediğin bir ayrıntıda yanlış çıkar.

Değer, soru sayısında değil. Kararları kod yazılmadan önce yazıya dökmekte — böylece inşa süreci bunları dosya dosya sessizce yeniden karara bağlamaz.

When it fires

Triggers on a pitch, steps aside for a spec

Fires "I want to build X, let's start building" — greenfield, requirements only in your head.
Fires The idea is vague enough that three different builds would all technically satisfy it.
Steps aside You already supplied a detailed spec, PRD, or design doc.
Steps aside Small, well-bounded change to existing code — a bug fix, one endpoint.
Ne zaman devreye girer

Devreye girer "X'i yapmak istiyorum, hadi başlayalım" — sıfırdan başlanan, gereksinimleri sadece kafanda olan işler.

Devreye girmez Mevcut koda küçük, sınırları belli bir değişiklik — bir hata düzeltmesi, tek bir endpoint.
The interview, in six steps

  1. Scope the batch. Question count is estimated from project size, not fixed at any round number.

  2. Ask in one batch, grouped by category — not one question at a time.

  3. Let answers stay loose. "You decide" and skipped items are fine — they become visible assumptions, not silent ones.

  4. Compile a spec document from everything gathered.

  5. Confirm the spec with a short "here's what I heard" pass before touching code.

  6. Carry the spec forward as the primary reference for the whole build, not just the first message.

Altı adımda interview

  1. Soru grubunun kapsamını belirle. Soru sayısı, sabit bir yuvarlak sayı değil, proje büyüklüğüne göre tahmin edilir.

  2. Tek seferde, kategorilere göre grupla sor — tek tek değil.

  3. Cevapların gevşek kalmasına izin ver. "Sen karar ver" ve atlanan sorular sorun değil — bunlar gizli değil, görünür varsayımlar haline gelir.

  4. Toplanan her şeyden bir spesifikasyon dokümanı derle.

  5. Koda dokunmadan önce spesifikasyonu onaylat — kısa bir "ben şunu anladım" turu ile.

  6. Spesifikasyonu ileri taşı — sadece ilk mesaj için değil, tüm inşa sürecinin ana referansı olarak.

The output

A spec file, not a chat transcript

Written to a real file so it survives context compaction and stays the reference for the whole build — not just the opening message.

SPEC.md
# Recipe Sharing App — Spec

## Goal
Let home cooks share recipes and rate each other's dishes.

## Users
Home cooks, casual, mobile-first, no account required to browse.

## Features (v1 / must-have)
- Post a recipe with photo, ingredients, steps
- Rate 1–5 stars, one rating per user per recipe

## Explicitly out of scope
- Meal planning, grocery lists, monetization

## Open assumptions
- Moderation: none specified — flagged for review before launch

# Tarif Paylaşım Uygulaması — Spesifikasyon

## Amaç
Ev aşçılarının tarif paylaşıp birbirinin yemeklerini puanlamasını sağlamak.

## Kullanıcılar
Ev aşçıları, gündelik kullanım, mobil öncelikli, göz atmak için hesap gerekmiyor.

## Özellikler (v1 / olmazsa olmaz)
- Fotoğraf, malzeme ve adımlarla tarif paylaşma
- 1–5 yıldız puanlama, kullanıcı başına tarif başına bir puan

## Kesinlikle kapsam dışı
- Yemek planlama, market listesi, gelir modeli

## Açık varsayımlar
- Moderasyon: belirtilmedi — lansmandan önce gözden geçirilmek üzere işaretlendi
Common mistakes it prevents
＋

Four shallow questions and done. Stack, features, deploy target, database is a developer checklist — it skips goals, users, design, edge cases, and non-goals.

＋

An easy skip-hatch offered upfront. Skipping should be the user's call, not the path of least resistance the agent nudges toward.

＋

One question at a time. Burns turns, loses attention. Batch by category instead.

＋

Answers left in chat only. If it's not written to a file, it doesn't survive to the implementation phase.

Önlediği yaygın hatalar
＋

Dört yüzeysel soru sorup bitirmek. Yığın, özellikler, deploy hedefi, veritabanı bir geliştirici kontrol listesidir — amaç, kullanıcılar, tasarım, uç durumlar ve kapsam dışılığı tamamen atlar.

＋

Baştan kolay bir "soruları atla" kapısı sunmak. Atlamak kullanıcının tercihi olmalı, ajanın yönlendirdiği en kolay yol değil.

＋

Soruları tek tek sormak. Turları tüketir, dikkati dağıtır. Bunun yerine kategoriye göre grupla.

＋

Cevapları sadece sohbette bırakmak. Bir dosyaya yazılmazsa implementasyon aşamasına kadar hayatta kalmaz.

Install

Drop the skill folder into your skills directory:

skills/requirements-interview/ → ~/.claude/skills/requirements-interview/

Global once installed — fires automatically in any project when a pitch, not a spec, lands in the chat.
